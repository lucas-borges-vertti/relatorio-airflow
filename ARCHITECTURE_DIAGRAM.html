💡 Padrão utilizado: Request-Response Assíncrono com Polling
  • Frontend obtém ID de requisição imediatamente (202 Accepted)
  • Polling a cada 5 segundos para acompanhar status
  • Processamento ocorre em background via Airflow
  • Callback notifica quando pronto

🏗️ 5 Tiers da Arquitetura

🖥️ TIER 1: Frontend (React)

React
JavaScript

Responsabilidades:

  • Detecta período ≥30 dias
  • Abre modal "Relatório Assíncrono"
  • POST /api/reports/async
  • Polling GET /api/reports/{id}/status
  • Exibe progresso e download links

🔌 TIER 2: API REST (NestJS)

NestJS 10.x
TypeORM
Node.js

Responsabilidades:

  • Recebe requisições HTTP
  • Valida DTOs (class-validator)
  • Persiste em PostgreSQL
  • Dispara DAGs no Airflow
  • Fornece status em tempo real
  • Recebe callbacks do Airflow

🚀 TIER 3: Orquestração (Apache Airflow)

Airflow 2.8.1
Python
PythonOperator

DAG: report_generation_dag

  • Task 1: extract_data → Consulta Oracle
  • Task 2: transform_data → Gera PDF/CSV
  • Task 3: send_email → Entrega ao usuário
  • Task 4: callback → Notifica NestJS

💾 TIER 4: Dados

PostgreSQL
Oracle DB

Responsabilidades:

  • PostgreSQL: Metadados (velog_reports_async)
  • Oracle: Fonte de dados (múltiplos schemas)
  • Isolamento por cliente via schema mapping

📦 TIER 5: Storage

OCI Object Storage
Cloud Storage

Responsabilidades:

  • Armazena PDFs gerados
  • Armazena CSVs gerados
  • Fornece URLs de download (1 hora de validade)

⏱️ Timeline de Execução

T+0s

Usuário Clica "Solicitar Relatório"

Frontend valida período (≥30 dias) e abre modal.

Frontend
T+1s

POST /api/reports/async

Frontend envia requisição com payload completo para NestJS.

Network (1 seg de latência)
T+2s

NestJS Valida e Persiste

API valida DTO (class-validator), cria registro em PostgreSQL com status=PENDING.

NestJS (1 seg de processamento)
T+3s

NestJS Dispara Airflow DAG

Axios faz POST para Airflow API, disparando report_generation_dag.

NestJS (1 seg de processamento)
T+4s

202 Accepted + requestId

NestJS retorna resposta imediata com ID para polling.

Frontend (status = PENDING)
T+5s

Airflow Inicia Task 1: extract_data

Scheduler Airflow começa a executar primeira task (extração Oracle).

Airflow (status: PROCESSING)
T+5m~10m

extract_data Concluída

Oracle retornou dados. Resultados salvos em XCom (memória).

Airflow Task 1
T+10m~11m

transform_data: Gera PDF e CSV

ReportLab formata PDF conforme template do cliente. CSV gerado com UTF-8-SIG.

Airflow Task 2 (30-60 seg)
T+11m~12m

PDF/CSV Salvos em OCI Storage

Arquivos enviados para OCI, object_keys recuperadas (ex: "reports/uuid/file.pdf").

Airflow Task 2
T+12m~12m30s

send_email: Email com Anexos

Airflow conecta SMTP, envia email para usuario_email com PDF/CSV e URLs download.

Airflow Task 3 (10 seg)
T+12m40s

callback: Notifica NestJS

Airflow faz PATCH /api/reports/{id}/status com status=COMPLETED e object_keys.

Airflow Task 4 (5 seg)
T+12m45s

NestJS Atualiza PostgreSQL

Registro agora tem status=COMPLETED, object_key_pdf e object_key_csv preenchidos.

NestJS (status: COMPLETED)
T+13m

Próxima Chamada de Polling do Frontend

Frontend faz GET /api/reports/{id}/status (a cada 5 seg) e recebe status=COMPLETED.

Frontend (vê ✅ Pronto!)
T+13m+

Usuário Recebe Email e Acesso

Email com PDF/CSV entregue. URLs de download (1 hora) disponíveis no Portal.

Frontend → Usuário (😊 Sucesso!)
